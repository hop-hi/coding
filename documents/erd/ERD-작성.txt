2. 데이터 베이스 설계 <- 요구사항
  - 데이터 베이스 설계 or 시스템 설계 or 서비스 설계 ... 등등...
  - 모두 시작은  "요구사항 수집 -> 분석 -> 정의" 를 통해서 시작 함.
  - 시스템/서비스 설계가 데이터 베이스 설계를 포함하는 경우가 대부분
  - 데이터 베이스만 별도로 설계 하는 경우는 거의 없음.

3. 전통적인 모델링 (RDB : 관계형 DB)
  - 스키마(내부:물리적-DBA/외부:사용자-개발자/개념-설계자)가 있음
    * 지식을 표상하는 구조 / 프레임 / 선험적 지식(경험이 없어도.. Not 배경지식)
    * 데이터베이스의 논리적 구조/제약조건 -> 정의/저장
    * Oracle, MySQL, PostGreSql...
  - 개념 모델링( Conceptual )
    * 개체(핵심) 도출
    * 관계 도출 / 개체 관계도
  - 논리 모델링( Logical )
    * 상세 속성 정의
    * 정규화 
  - 물리 모델링( Physical )
    * DB 개체 정의
    * 테이블 / 인텍스

3. NoSQL ( Not 관계형 )
  - Cassandra, MongoDB, Redis....
  - 스키마가 없음

4. 모델링 (관계형 데이터베이스 모델링)
  - 모델링 ? 현재 상황에서 "CREATE TABLE XXXXX ....." 할 수 있는가?
  - 할 수 없다면, 무엇이 부족한 건가?
  - 더 정확하고, 명확한 테이블 구조와 관계에 대한 정보(정의)가 필요함
  - 완벽하게 테이블을 생성해 낼 수 있는 구조를 정의 하는 작업 : 모델링
  - 모델링에는 서로 이해 할 수 있는 언어가 있어야 함.(공통의)
  - 글 / 그림 ??
  - 둘 다 사용 함.
  - 글 : 문서(템플릿) / 그림(다이어그램)

5. 요구사항을 정의해야 -> 관계형 데이터 베이스 모델링을 하게 됨. 
  - 만약 6번 작업을 지금 당장 한다면???? 
  - 요구사항을 수집->분석->정의 하는 과정은 결국 필요 함.
  - 요구사항을 수집 하는 방법
    : 기존 문서, 사용자 인터뷰 방식 
  - 요구사항을 분석 하는 방법
    : 필요한 정보(요소)를 식별하고, 요소들간의 관계와 흐름을 표현 
  - 요구사항을 정의 하는 방법
    : 다음작업(데이터베이스 모델링, 객체 모델링 .... 등....)을 위한 결과물을 만들어 냄.
      ex) 개체 후보들과 관계, 속성 후보들을 정리한 "엔티티 정의서" <- 데이터베이스(관계형) 모델링을 위함
